1. Какие три основные области работы с файлами в Git (рабочая папка, индекс, репозиторий) и чем они отличаются?
Рабочая папка — это ваши текущие файлы. Индекс — буфер, куда вы добавляете изменения для коммита. Репозиторий — база данных с полной историей всех коммитов.

2. Что происходит при выполнении команд:
git add
git commit
git push
git add помещает изменения из рабочей папки в индекс. git commit фиксирует изменения из индекса в локальный репозиторий. git push отправляет ваши коммиты из локального репозитория в удалённый (например, на GitHub).

3. Почему важно писать информативные сообщения к коммитам а не просто "исправление" или "update"?
Информативные сообщения объясняют, зачем были сделаны изменения, а не просто констатируют факт. Это помогает команде и вам в будущем быстро понять историю и цель каждого изменения без чтения кода.

4. Чем отличается git add имя_файла от git add .? В каких случаях лучше использовать каждый вариант?
git add имя_файла добавляет в индекс только конкретный файл, давая точный контроль. git add . добавляет все новые и изменённые файлы в текущей папке. Первый вариант лучше для точечных коммитов, второй — для быстрого добавления всей проделанной работы.

5. Что показывает команда git status и как меняется ее вывод на разных этапах работы?
Команда git status показывает состояние рабочей папки и индекса: какие файлы изменены, но не добавлены в индекс; какие добавлены и готовы к коммиту; какие не отслеживаются Git. Её вывод меняется по мере выполнения git add и git commit.

6. Для чего нужна команда git remote add origin URL и что означает слово "origin"?
Эта команда связывает ваш локальный репозиторий с удалённым репозиторием по указанному URL. Слово "origin" — это стандартное короткое имя (псевдоним) для этого удалённого репозитория.

7. Зачем нужен флаг -u в команде git push -u origin main при первой отправке изменений?
Флаг -u устанавливает связь между вашей локальной веткой main и удалённой веткой origin/main. После этого в дальнейшем можно использовать просто git push или git pull, и Git будет знать, с какой удалённой веткой работать.

8. Что такое ветка (branch) по умолчанию и какая у вас создалась (main или master)? Почему это важно?
Ветка по умолчанию (чаще main, раньше master) — это основная ветка, в которую обычно вливаются новые функции. Её имя важно, так как она считается стабильной версией проекта, и многие инструменты ориентируются на неё.

9. Как просмотреть историю коммитов и какую информацию о каждом коммите вы видите?
Историю коммитов просматривают командой git log. Для каждого коммита вы видите его уникальный хеш, автора, дату и сообщение коммита.

10. Что произойдет, если сделать изменения в файле, но не выполнить git add перед git commit?
Если сделать изменения в файле, но не выполнить git add перед git commit, эти изменения не попадут в коммит. Git проигнорирует их, так как коммит создается только из тех изменений, которые были предварительно добавлены в индекс (staging area).
